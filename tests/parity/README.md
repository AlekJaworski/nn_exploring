# Parity tests vs R `mgcv`

This directory enforces and tracks how closely `mgcv_rust` reproduces the
output of R's `mgcv::gam()` on a battery of cases.

## Layout

```
tests/parity/
├── README.md              # this file
├── schema.py              # Python dataclass mirror of the fixture JSON schema
├── conftest.py            # pytest fixtures, tolerance config, helpers
├── test_parity.py         # the parameterized parity test
├── cases.py               # the canonical battery of cases (names + params)
└── fixtures/
    ├── 1d_gaussian_smooth_n500_k10.json
    ├── 1d_gaussian_wiggly_n500_k20.json
    ├── ...                # one JSON per case
    └── .gitkeep
```

Fixtures are generated by `scripts/python/generate_parity_fixtures.py`,
which calls mgcv via `rpy2` (same pattern as
`libraries/r_fitting/src/r_fitting/r_model.py` in the neighbourhoods
repo — `mgcv.bam(formula, data, method, family)`, no k-tuning loop).
Fixtures are checked in so the test run only needs Python +
`mgcv_rust`. Regenerate with:

```bash
python scripts/python/generate_parity_fixtures.py
# or just one case
python scripts/python/generate_parity_fixtures.py 1d_gaussian_smooth_n500_k10_cr
```

The generator requires R + the `mgcv` package + `rpy2` in the active
Python environment.

## Three bars of agreement

Every fixture is checked against `mgcv_rust` at three levels. **Bar A**
gates merges; **B** and **C** are tracked, ratchet up over time as we
close gaps.

| Bar | What we assert | Default tolerance | Status |
|---|---|---|---|
| **A** | `predict()` matches at training, held-out, and extrapolation points | `rtol=1e-3`, `atol=1e-6` | **block on failure** |
| **B** | β coefficients, vcov diagonal, EDF per smooth, deviance, scale match | `rtol=1e-3` for β, `rtol=5e-3` for vcov diag, `atol=0.05` for EDF | track + report |
| **C** | λ smoothing parameters match | `rtol=5%` | track + report |

The full results matrix is emitted as `tests/parity/results.json` and
`tests/parity/results.md` after every run, so ratcheting is just a
config change.

## Fixture schema (v1)

One JSON file per case. All arrays serialized as nested lists.

```jsonc
{
  "schema_version": 1,
  "name": "1d_gaussian_smooth_n500_k10",
  "description": "Smooth 1D sin curve, Gaussian noise, k=10 cr-spline",

  "metadata": {
    "mgcv_version": "1.9-1",
    "r_version": "R 4.3.3",
    "generated_at": "2026-04-28T12:00:00Z"
  },

  "inputs": {
    "seed":     42,
    "n":        500,
    "d":        1,
    "k":        [10],            // length d
    "bs":       ["cr"],          // length d, one of "cr" or "bs"
    "family":   "gaussian",      // "gaussian" | "binomial" | "poisson" | "gamma"
    "link":     "identity",      // "identity" | "log" | "logit" | "inverse"
    "method":   "REML",          // "REML" | "GCV.Cp"
    "weights":  null,            // null = unit weights, else length-n list
    "x_train":  [[...], ...],    // n × d
    "y_train":  [...],           // length n
    "x_test":   [[...], ...],    // m × d, in-range held-out points
    "x_extrap": [[...], ...]     // p × d, points outside training range
  },

  "mgcv_output": {
    "beta":                 [...],          // length p_total
    "vcov":                 [[...], ...],   // p_total × p_total (Bayesian, vcov.gam default)
    "lambda":               [...],          // length d (one per smooth)
    "edf_per_smooth":       {"x0": 8.2, "x1": 5.1, ...},   // by smooth name
    "edf_total":            9.2,            // including intercept
    "deviance":             5.234,
    "scale":                0.04,           // φ
    "n_iter":               5,              // outer optimizer iterations

    "predictions_train":    [...],          // length n, response scale
    "predictions_test":     [...],          // length m, response scale
    "predictions_extrap":   [...],          // length p, response scale

    "predictions_train_se": [...],          // length n, response-scale SE
    "predictions_test_se":  [...]           // length m, response-scale SE
  }
}
```

### Field-by-field semantics

`inputs.bs[i]` = mgcv's basis spec passed to `s(...)`. Currently `"cr"`
(cubic regression) and `"bs"` (B-spline). More can be added as `mgcv_rust`
grows.

`inputs.weights` = `weights` arg to `gam()`. `null` means no weights
argument given; non-null is a length-n vector of `prior weights` (note:
not IRLS weights — those are derived during fitting).

`inputs.x_test` / `inputs.x_extrap` are deliberately split. Bar A applies
to both, but extrapolation usually has looser tolerance (lambda
disagreement amplifies away from data).

`mgcv_output.beta` is the full coefficient vector in the order mgcv lays
out: intercept first, then each smooth's basis columns in declared order.
`mgcv_output.vcov` matches that ordering. `predictors_info` (start/end
column indices per smooth) is derivable from the `.smooth` slot on the
fitted gam object — emitted alongside if needed by the test (TBD).

`mgcv_output.scale` is φ (`gam$sig2` for Gaussian, `gam$scale` for
non-Gaussian). For Bar B comparisons we may rescale vcov by φ to isolate
geometry from scale.

`mgcv_output.predictions_*` is on the **response scale** (after inverse
link). `mgcv_output.predictions_*_se` is the **response-scale standard
error** (mgcv computes these directly via `predict(..., se.fit=TRUE)`).

`mgcv_output.edf_per_smooth` is a dict so it's robust to smooth ordering
changes. Names are mgcv's labels: `"s(x0)"` → key `"x0"` (we strip the
`s(...)` wrapper in the generator).

## What's *not* in the fixture

- IRLS weights at convergence (not generally needed for parity; can be
  added if a Bar B failure traces back to weight differences).
- `Vp` vs `Vc` vs `Ve` distinctions — we use `vcov.gam()` default
  (Bayesian, `Vp`). Frequentist `Ve` could be emitted later if needed.
- `gam.check()` diagnostics — those are sanity, not parity.
- Plot data (e.g., `plot.gam` partial smooths). Could be added if we
  start verifying smooth-by-smooth shapes.

## Versioning

`schema_version: 1` at the top. If a field is added, bump to 2 and update
both the R generator and the pytest loader. Old fixtures stay readable
because we tag them with the version they were generated under.
